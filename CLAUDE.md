# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A minimal, production-ready starter for building SaaS apps: Next.js 15 (App Router) fullstack, Tailwind CSS v4, i18n via next-intl, TanStack Query for client-side server-state, client-side forms via react-hook-form + Zod, toast feedback via sonner, email/password auth via Better Auth, TypeScript strict, PostgreSQL via Drizzle ORM, Zod validation, Playwright E2E, Docker multi-stage, GitHub Actions CI. README and code comments are in Portuguese; keep that convention when editing existing comments.

## Commands

```bash
npm run dev              # dev server (hot reload)
npm run build / start    # production build / serve
npm run lint              # next lint
npm run typecheck         # tsc --noEmit
npm run ci:verify         # lint + typecheck (what CI's "quality" job runs)

npm run db:generate       # generate SQL migration from src/db/schema.ts
npm run db:migrate        # apply pending migrations (tsx scripts/migrate.ts)
npm run db:studio         # Drizzle Studio (visual DB browser)

npm run test:e2e          # Playwright, auto-starts the app via webServer
npm run test:e2e:ui       # Playwright interactive UI mode
```

Running a single Playwright test/file: `npx playwright test e2e/api.spec.ts` or `npx playwright test -g "rejeita payload inválido"`.

Local DB setup (required before `dev` or `test:e2e` will work against real data):

```bash
cp .env.example .env
docker compose up -d db
npm run db:generate && npm run db:migrate
```

Multiple projects from this template running side by side must use distinct `DB_PORT`/`DB_NAME` in `.env` (see comments in `.env.example`); `docker-compose.yml` reads these and namespaces the data volume by project folder name to avoid collisions.

## Architecture

- **Schema-first DB layer**: `src/db/schema.ts` is the single source of truth for the database. Migrations in `drizzle/` are generated from it (`db:generate`) — never hand-edit generated SQL. `src/db/index.ts` exposes a singleton Drizzle client (`db`); the singleton is cached on `globalThis` in non-production to survive dev hot-reload without exhausting the connection pool. For serverless deploys (Vercel/Lambda), swap in a pooler (PgBouncer/Neon/Supabase Pooler) and set `max: 1`.
- **Multi-tenancy is baked into the schema**: `organizations` is the tenant root; every domain table (see `todos`) has a required `organization_id` FK with `onDelete: cascade`. **`src/lib/crud.ts`'s `resolveOrganizationId()` intentionally still uses the `getDefaultOrgId()` placeholder (`src/lib/organization.ts`) unconditionally, session or not** — this is a deliberate decision (spec 0003), not an oversight: the homepage's todo list/form (spec 0005) is public with no sign-in, and gating `/api/todos` would break that demo. Auth exists as real, working infrastructure (see the Better Auth bullet below) without gating any existing resource yet. A *new* resource that should be session-scoped calls `getSession()`/`requireSessionOrRedirect()` (from `src/lib/auth.ts`) directly in its own route/page — `src/app/api/account/route.ts` and `src/app/[locale]/account/page.tsx` are the reference pattern for that, not `crud.ts`.
- **API routes for a resource are built with `src/lib/crud.ts`'s `createCrudHandlers()` factory**, not hand-written per route. It takes a Drizzle table, its Zod insert/update schemas, and allow-lists for sortable/filterable columns, and returns `{ collection: { GET, POST }, item: { GET, PATCH, DELETE } }` — every query it builds is scoped by `organizationId` with no way to opt out, and get/update/delete all `404` (never `403`) on a missing or cross-tenant id, so existence isn't leaked. Every response uses a `{ data, ... }` envelope (`{ data, page, pageSize, total }` for lists). `src/app/api/todos/` is the reference pattern to copy for a new resource: `_crud.ts` builds the config once (`_`-prefixed so Next.js doesn't route it), `route.ts` re-exports `todosCrud.collection`, `[id]/route.ts` re-exports `todosCrud.item`. A route file may **only** export HTTP method handlers and a small set of route-config fields — don't add other exports to it (re-export shared values like Zod schemas from their own module instead, e.g. `src/lib/schemas/todos.ts`).
- **Env validation fails fast at boot**: `src/lib/env.ts` parses `process.env` through a Zod schema at import time; any module importing `env` will throw immediately if `DATABASE_URL`/`BETTER_AUTH_SECRET`/`BASE_URL` is missing/invalid rather than failing later with an unclear error. Add new required env vars here, not via ad-hoc `process.env.X` reads. `BETTER_AUTH_SECRET` (min 32 chars) and `BASE_URL` are both required by `src/lib/auth.ts`'s `betterAuth({ secret, baseURL, ... })` — without `baseURL` set to the app's actual public URL, Better Auth rejects every request with `Invalid origin: ...` (it has no way to know its own origin is trustworthy). Whichever port the app is actually reached on (`APP_PORT` in Docker, or your local dev port) must match `BASE_URL`'s port, or that same error resurfaces. The Dockerfile sets placeholder values for both at build time (never used for anything real — same pattern as the existing fake `DATABASE_URL`); the real values come from `docker-compose.yml`'s `app.environment`, sourced from your own `.env`.
- **Healthcheck (`/api/health`) is load-bearing infrastructure**: it's polled by the Docker `HEALTHCHECK`, the docker-compose dependency graph, and Playwright's `webServer.url` (tests won't start running until it responds), and the homepage also exercises the same DB round-trip to render a "conectado/indisponível" status. Don't change its response shape (`{ status, db }`) without checking these dependents.
- **Migrations run in three different places** with the same script (`scripts/migrate.ts`): locally (`npm run db:migrate`), in CI before the E2E job builds/tests, and automatically in the Docker entrypoint (`docker-entrypoint.sh`) on container boot — set `SKIP_MIGRATIONS=true` to run migrations as a separate release step instead in a multi-replica deployment.
- **Dockerfile is a 3-stage build** (`deps` → `builder` → `runner`) producing a `next.config.ts` `output: "standalone"` image (~150MB, non-root `nextjs` user). The build stage sets a fake `DATABASE_URL` only so Zod env validation passes during `next build`; no real DB connection happens at build time. The runner stage selectively copies only the node_modules needed to run `scripts/migrate.ts` at boot (drizzle-orm, postgres, tsx, esbuild, get-tsconfig, resolve-pkg-maps) — if migrate.ts's dependency graph changes, this copy list needs updating too.
- **Styling is Tailwind CSS v4, CSS-first**: there is no `tailwind.config.js` — `src/app/globals.css` does `@import "tailwindcss"` and defines design tokens in an `@theme` block (`--color-bg`, `--color-surface`, `--color-text`, `--color-muted`, `--color-accent`, `--color-border`), which Tailwind turns into matching utilities (`bg-surface`, `text-muted`, `border-border`, …) automatically. `postcss.config.mjs` wires in `@tailwindcss/postcss`. Tailwind's Preflight reset is active, so headings/paragraphs have no default browser margin/size — set spacing and type scale explicitly with utility classes (see `src/app/[locale]/page.tsx`).
- **All pages live under `src/app/[locale]/`, i18n via next-intl**: there is no `src/app/layout.tsx` — `src/app/[locale]/layout.tsx` is the actual root layout (it renders `<html lang={locale}>`/`<body>`). A root `middleware.ts` resolves the active locale (cookie → `Accept-Language` → `pt-BR` default) and redirects unprefixed requests; its matcher explicitly excludes `/api/**`, so route handlers are untouched by locale routing and stay at their current unprefixed paths. Locales are defined once in `src/i18n/routing.ts` (`pt-BR`, `en`); message catalogs live in `src/i18n/messages/{locale}.json`. Every user-facing string goes through `getTranslations`/`t(...)`, never a hardcoded literal — add new keys to **both** locale files. `src/i18n/navigation.ts` exports locale-aware `Link`/`usePathname`/`useRouter`; use those instead of `next/link`/`next/navigation` in anything under `[locale]`.
- **The root layout wraps `children` in `NextIntlClientProvider`** (messages from `getMessages()`) — this is required, not optional. Any Client Component that uses `src/i18n/navigation.ts`'s `Link`/`usePathname`/`useRouter` throws `"No intl context found"` without it, even if that component never calls `useTranslations`/`useLocale` itself (`src/components/locale-switcher.tsx` hit exactly this before the provider was added). Prefer passing the active `locale` down as a prop from a Server Component instead of calling `useLocale()` in new Client Components — one less thing depending on the provider being wired correctly.
- **TanStack Query** (`src/components/query-provider.tsx`, mounted in `src/app/[locale]/layout.tsx`) is for client-side server-state: interactive lists (search, pagination, sorting), mutations, polling — anything fetching from `src/app/api/**` after the initial render. It is **not** a replacement for the existing pattern of querying `db` directly in a Server Component for a page's first render — keep using that for data needed at initial render. `src/components/todo-form.tsx` is the reference implementation (`useQuery` for the list, `useMutation` for create).
- **Client-side forms pair `react-hook-form` with the same Zod schema the API route validates with** — never a second, hand-maintained copy of the validation rules. A resource's create/update schema lives in `src/lib/schemas/<resource>.ts` (e.g. `createTodoSchema`), imported by both the resource's `_crud.ts` (see the `createCrudHandlers` bullet above) and the form component; it is **not** re-exported from the route file itself (`route.ts` files may only export HTTP method handlers and a small set of route-config fields — Next.js rejects anything else at build time). `src/lib/forms/use-zod-form.ts` wraps `useForm` with `zodResolver` pre-wired; `src/lib/forms/server-errors.ts`'s `mapZodErrorsToForm` maps a `400` response's `details.fieldErrors` (the existing `parsed.error.flatten()` shape) onto the form via `setError`, called from the mutation's per-call `onError` (the mutation hook's own `onError` only toasts non-validation failures, so a validation error gets exactly one thing: the inline field message — never also a generic toast). `@hookform/resolvers` is pinned to `^4` (not the latest `^5`) — v5's dual Zod v3/v4 typing doesn't structurally match this project's installed Zod 3.25.x and fails to typecheck.
- **Toast feedback is `sonner`, wrapped by `src/lib/toast.ts`**: call `showSuccess(message)`/`showError(message)` from there — don't import `sonner` directly in a component, so the library stays swappable from one place (same precedent as `src/db/index.ts`, `src/lib/env.ts`). Its `<Toaster/>` is mounted once in `src/app/[locale]/layout.tsx`, themed with the same `--color-surface`/`--color-border`/`--color-text` tokens as everything else. `src/components/health-check-button.tsx` is the reference call site (re-checks `/api/health` and toasts the result) — wire new mutations' `onSuccess`/`onError` the same way, and don't also toast a field-validation error that's already shown inline on a form.
- **Auth is Better Auth** (`src/lib/auth.ts`), Drizzle-adapter-backed against `user`/`session`/`account`/`verification` tables in `src/db/schema.ts` (column set verified against the installed package, not guessed — see the file's own comment). `advanced.database.generateId: "uuid"` keeps ids consistent with every other table's `uuid().defaultRandom()`. One project-specific addition: a nullable `organizationId` on `user`, added via `additionalFields`. **Org provisioning is lazy**, not a Better Auth `databaseHooks` callback: `getSession(headers)` in `src/lib/auth.ts` creates an `organizations` row and backfills `user.organizationId` the first time it resolves a session for a user without one — since `/account` (the first page after sign-up) always calls it, the org exists before the user can do anything else. `requireSessionOrRedirect(headers, locale)` wraps it for Server Components/pages (redirects to `/sign-in` via `src/i18n/navigation.ts`'s locale-aware `redirect()` — note that a bare, non-`return`ed call to a `never`-returning function doesn't always narrow control flow in this TS setup, so `requireSessionOrRedirect` follows the `redirect(...); throw new Error("unreachable");` pattern to force it); route handlers call `getSession(request.headers)` directly and `401` themselves (see `src/app/api/account/route.ts`). Client-side: `src/lib/auth-client.ts`'s `authClient` (`signIn`/`signUp`/`signOut`/`useSession`) — `src/components/auth-nav.tsx` is the reference for reading session state in a Client Component. `better-auth`'s peer deps forced upgrading `drizzle-orm`/`drizzle-kit` to `^0.45.2`/`^0.31.10` (from `^0.36.4`/`^0.28.1`); that bump introduced a stricter `.from()` type guard that a fully generic table parameter can't satisfy, which is why `src/lib/crud.ts` casts `table` to `any` at its `.from()` call sites specifically (documented inline).
- **Linting is ESLint v10 with flat config** (`eslint.config.mjs`, not `.eslintrc.json` — that file no longer exists). This config does **not** use `eslint-config-next`'s bundled `core-web-vitals` export — as of this writing, that pulls in `eslint-plugin-react` and `eslint-plugin-jsx-a11y`, neither of which has been updated for ESLint v10 yet (`eslint-plugin-react` actively crashes: `context.getFilename is not a function`). Instead, `eslint.config.mjs` uses `@next/eslint-plugin-next`'s own standalone `configs["core-web-vitals"]` (Next-specific rules only, no React/a11y linting, no ESLint version constraint of its own) plus `eslint-config-next/typescript` (backed by `typescript-eslint` v8, which does declare `eslint: ^10.0.0` support). **Known gap**: general React best-practice and accessibility (jsx-a11y) lint rules are currently unavailable — re-add `eslint-config-next/core-web-vitals` once its upstream deps catch up to ESLint v10. `eslint.config.mjs` also sets `@typescript-eslint/no-unused-vars`'s `argsIgnorePattern`/`varsIgnorePattern` to `^_` — the convention for an intentionally-unused parameter (e.g. a placeholder kept for a stable call signature) is a leading underscore, not an inline disable comment.
- **CI (`​.github/workflows/ci.yml`)** is a 3-job pipeline gated in sequence: `quality` (lint+typecheck) → `e2e` (real `postgres:17-alpine` service container, migrate, production build, Playwright) → `docker` (build+push to GHCR, only on push to `main`). Playwright runs against a production build in CI (`npm run start`) but against `npm run dev` locally, per `playwright.config.ts`.

## TypeScript

`strict` + `noUncheckedIndexedAccess` are both on — array/object index access returns `T | undefined`, so non-null assertions or explicit checks are required where the codebase already does this (e.g. `created!.id` after a Drizzle `.returning()` where a row is known to exist).

**TypeScript is pinned to `^5.9.3`, not the latest 7.x** — `typescript-eslint` (which `eslint-config-next/typescript` depends on) throws a hard error on TS 7.0 (`typescript-eslint does not support TS 7.0`) rather than just warning, and even at its latest release its peer range caps below TS 6.1. This isn't a style preference; TS 7 breaks the lint pipeline outright (and very likely was the cause of a `next build`/`next dev` "build worker exited with code: 1" crash, since Next's build runs ESLint internally). Don't bump `typescript` past the 5.x line until `typescript-eslint` publishes real support for newer majors — check its peer `typescript` range first.

Also note: with Zod v4 (see the `zod` version in `package.json`), a generic wrapper like `src/lib/forms/use-zod-form.ts` needs its schema parameter constrained as `z.ZodType<TFieldValues, TFieldValues>` (parameterized by the field-values type first) — a bare `Schema extends z.ZodType` constraint defaults both of Zod v4's `Output`/`Input` type parameters to `unknown`, which silently breaks `z.infer<Schema>` inference through the generic and cascades into type errors in every form that uses it.

## Agent team workflow

Feature work in this repo is done by a team defined in `specs/0001-creation-of-the-teammates/`: you (the orchestrating session) act as **Lead**, delegating to the `backend`, `frontend`, and `test` subagents in `.claude/agents/`. This governs every spec in `specs/`, present and future.

- `specs/` is the source of truth. Read the relevant spec (`requirements.md`, `design.md`, `tasks.md`) before planning or delegating anything.
- As Lead, draft an implementation plan for the spec — task breakdown, which teammate owns what, order of work, and any shared-type changes — and **present it to the user for explicit approval before any teammate touches a file.** No exceptions for specs that look small.
- Only after explicit approval: dispatch `backend` first (it updates shared types before finishing routes, so nothing downstream is blocked), then `frontend` and `test` in parallel against the now-approved contract.
- Shared types are Backend's contract with everyone else — Drizzle-inferred row types in `src/db/schema.ts` plus any request/response shapes for its routes. Frontend and Test consume them; they never redefine them.
- When a spec is ambiguous, the responsible teammate checks existing project conventions for precedent first and only escalates to you if it's still unclear — never guesses.
- Each teammate has a matching skill in `.claude/skills/` (`backend`, `frontend`, `test`) with this repo's concrete stack patterns (schema/migration/route steps, Server Component + Tailwind conventions, Playwright test patterns) — reach for it before writing that kind of code by hand.
